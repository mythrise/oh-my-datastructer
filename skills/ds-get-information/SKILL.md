# /ds:get-information — 作业拆解与方法检索

将作业拆解为清晰的功能模块，为每个模块检索基础/进阶/创新三级实现方法。

## 核心理念

这个 skill 的读者是**想要了解自己项目究竟是什么**的学生。输出不是给专家看的技术文档，而是一份让学生建立心智模型的**架构蓝图**。

**两个并列目标：**
1. 让用户理解"这个作业由什么零件组成，每个零件做什么"
2. 为每个零件提供多级实现方案，便于用户选择创新方向

## 执行流程

### Step 1: 读取项目上下文

读取 `.pipeline/memory/project_truth.md` 获取：
- 作业名称和类型（topic_id）
- 原始需求描述
- 检测到的知识点

读取 `references/ds-taxonomy.md` 获取该类型的：
- 常见算法列表
- 评估指标
- 可视化类型

### Step 2: 分层拆解

将作业拆分为**功能层**：

| 层 | 职责 | 示例（哈夫曼） |
|----|------|----------------|
| **输入处理层** | 数据读取、格式解析、预处理 | 读取文件、字节流化 |
| **核心算法层** | 关键数据结构和算法 | 频率统计、树构建、编码生成 |
| **功能实现层** | 基于核心算法的应用功能 | 压缩、解压、分析 |
| **输出展示层** | 结果呈现、可视化、导出 | JSON导出、3D可视化、报告 |

### Step 3: 模块级拆分

对每一层拆分为具体的**功能模块**。每个模块按以下格式描述：

```markdown
#### 模块名称（技术术语）

**它是什么？**
[用一句话说清楚这个模块做什么，不用技术词汇]
例："这是一个统计员，负责数清楚文件里每个字符出现了多少次"

**它在技术上是什么形式？**
[说明这个模块在代码中的存在形式——函数？类？文件？]
例："在 Python 里，它是一个函数，接收字节数据，返回频率字典"

**输入什么？**
[用自然语言描述接收的数据]
例："接收一段原始字节数据（比如一个文件的全部内容）"

**输出什么？**
[用自然语言描述产出的结果]
例："返回一个字典：每个字节值 → 它出现的次数"

**为什么项目需要它？**
[如果没有这个模块会怎样]
例："如果不统计频率，就无法知道哪些字符常见、哪些罕见，哈夫曼编码的核心思想——给常见字符短编码——就无从谈起"

**基础实现方式：**
[教科书标准方法，含复杂度]
例："遍历一遍数据，用字典累加计数。时间 O(n)，空间 O(k)，k 为不同字符数。"

**进阶替代方案：**
[学术界常见的优化方法]
例："使用 collections.Counter（底层 C 实现，常数因子更小）。或对大文件分块并行统计后合并。"

**创新替代方案：**
[前沿/可供自己创新设计的方向]
例："自适应频率统计：滑动窗口动态更新频率，适用于非平稳数据源。可结合局部哈夫曼编码实现自适应压缩。"

**被谁调用？调用谁？**
[在项目流程中的位置]
例："被编码器调用 → 输出频率表传给树构建模块"
```

### Step 4: 标注核心模块

用 ⭐ 标注最多 1-3 个**核心模块**。

核心模块 = 项目最关键的算法在这里发生，这里的设计决策影响所有其他模块。

对核心模块，额外展开：

**算法逻辑（自然语言步骤）：**
写成"第一步……第二步……"的形式，用"想象你是这个函数，你会怎么做"的视角。

**关键设计决策：**
有哪些方案可选？每种方案的优缺点？

**可以加入的高级 idea：**
有哪些聪明的改进方向？能带来什么好处？

### Step 5: 学术方法检索

对每个核心模块，使用 WebSearch 检索：
- "数据结构 {模块关键词} 优化方法"
- "{algorithm_name} optimization techniques"
- "{ds_type} advanced implementation"

将检索到的方法补充到创新替代方案中。包含：
- 方法名称
- 核心思想（1-2句话）
- 复杂度改进
- 参考来源

### Step 6: 输出架构蓝图

输出到 `.pipeline/memory/project_truth.md`（追加）和终端：

```markdown
## 架构蓝图：{作业名称}
（AI 辅助编程的自然语言项目地图）

### 项目一句话描述
{用最简单的语言说这个项目做什么}

### 技术栈
{选用的语言、库、为什么适合}

### 项目流程总览
{首先……然后……最后……}

### 输入处理层
#### 模块 1.1：{模块名}
{按 Step 3 格式}

### 核心算法层
#### ⭐ 模块 2.1：{核心模块名}
{按 Step 3 + Step 4 扩展格式}

### 功能实现层
#### 模块 3.1：...

### 输出展示层
#### 模块 4.1：...

### 模块依赖关系
{用文本或 ASCII 图展示模块之间的调用关系}

### 推荐实现顺序
{先做什么、后做什么、为什么}
```

### Step 7: 更新管线状态

更新 `.pipeline/memory/orchestrator_state.md`：
```markdown
## 当前阶段
get-information → **已完成**

## 下一步
运行 /ds:plan 进行多轮规划讨论

## 架构蓝图摘要
- 共 {N} 个模块，{M} 层
- 核心模块：{列表}
- 检索到 {K} 个创新方法
```

## 注意事项

- 模块粒度要适中：太粗（整个编码器）无法指导实现，太细（单个变量）无法建立全局观
- 每个模块的三级方法必须有实质区别，不是简单换名字
- 创新方案不要求一定可行，但要说清楚思路和潜在好处
- 如果 WebSearch 失败，使用 `references/ds-taxonomy.md` 中的预置方法作为备选
